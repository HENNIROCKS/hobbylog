    🔀 Headless Root Redirect

    Visiting localhost:8000 previously showed Kirby's "page doesn't exist" error (no home template in a headless setup). A custom Kirby route redirects / to the SvelteKit frontend URL, reading the same env-aware config option used for CORS: 

    'routes' => [
  [
    'pattern' => '/',
    'action'  => function () {
      go(option('hb.frontend_url', 'http://localhost:5173'));
    },
  ],
],

    This also fixes the panel "Open" button on the site root — it previously looped back to /panel after the first iteration of this redirect pointed at /panel instead of the frontend.

    ⚙️ make backend Browser Auto-open 

    backend:
  composer start & sleep 1 && open http://localhost:8000/panel; wait
    • composer start & — PHP server starts in background
    • sleep 1 && open … — waits until server is ready, then opens /panel
    • wait — keeps the terminal occupied by the server process until Ctrl+C

    All on one Makefile line so it runs in a single shell — & on its own line would be orphaned when the subshell exits.

    System Pages — Kirby Collection

    Kirby's filterBy KQL does not support OR logic (chained calls are AND). Filtering to show only Home and Error in one section requires a Kirby Collection: 

    // site/collections/system-pages.php
return function ($kirby) {
  $children = $kirby->site()->children();
  return $children->filterBy('slug', 'home')
    ->add($children->filterBy('slug', 'error'));
};

    KQL gotcha: filterBy("intendedTemplate", ...) does not work for string comparison — intendedTemplate() returns a Template object. Even intendedTemplate.name failed in blueprint query context. Slug-based filtering is the reliable fallback.

    KQL gotcha: site.children.drafts returns nothing — drafts are not part of children(). Use site.drafts directly.

    🐛 Bug Fixes

    PaintProgress: entry count → amount sum

    PaintProgress.svelte calculated the painted percentage by dividing painted entries by total entries. A collection with one painted entry (×1) and one unpainted entry (×100) would show 50% — misleading. Fixed to sum amount, matching the index page logic. Also excludes amount === 0 entries from the denominator.

    Sort tie-breakers

    Amount sort and painted sort had no secondary key — items within a group appeared in arbitrary input order. Both now fall back to name A–Z: 

    if (sortBy === 'amount')  return b.amount - a.amount  || a.name.localeCompare(b.name, 'de');
if (sortBy === 'painted') return Number(a.painted) - Number(b.painted) || a.name.localeCompare(b.name, 'de');

    All localeCompare calls updated from 'en' to 'de' to handle German umlauts correctly.

    Mobile: tab nav wrapping

    Three tabs + badges could overflow on narrow screens. Changed flex gap-6 to flex flex-wrap gap-x-6 gap-y-0 — tabs reflow to a second line without breaking the underline indicator.

    Mobile: faction stats responsive widths

    Fixed widths w-40 / w-24 on the faction name and stats columns were squeezing the progress bar on mobile. Changed to w-28 sm:w-40 / w-16 sm:w-24.

    🚀 Mobile App Planning (Follow-up Project)

    A discussion at the start of the session explored turning the project into a proper mobile app. The conclusion: finish this project as a read-only hobby tracker first, then start a separate project with what was learned here.

    App options assessed

    🌐
    PWA — Low effort

    Add manifest.json + service worker to the existing SvelteKit build. Installs to home screen, works offline. ~80% of the app feel with ~20% of the effort. Recommended first step.

    📦
    Capacitor — Medium effort

    Wrap the SvelteKit build in Capacitor. Produces a real .ipa/.apk for App Store distribution. Minimal code changes — the web app shells inside the native wrapper.

    📱
    React Native — High effort

    Full rewrite in a native framework. Only justified if complex gestures, camera APIs, or platform-specific features are needed that the web cannot provide.

    The real constraint: editing

    Why not just wrap the current app in Capacitor now?
    The app shell is trivial. The hard part is write support — KQL is read-only, so editing miniatures from the app requires custom Kirby write endpoints, an auth flow, and image upload from camera → Kirby media. That work is the actual project, not the native wrapper.

    Follow-up project goals

    • Capacitor wrapper as a core design goal from day one
    • Custom Kirby PHP write API (mutations, not just queries)
    • Auth flow — app logs in as a Kirby user
    • Image upload: camera → Kirby media library
    • PWA first as a stepping stone before native packaging

    🚀 Deployment Automation

    Work in progress on branch feat/deployment-automation. The constraint: shared PHP hosting with no SSH and no Node.js runtime means the build must happen in CI and the output must be FTP'd.

    Scheduled build or webhook-triggered?
    Webhook-triggered. Content updates will be more frequent than code changes, so a scheduled build would mean waiting hours to see panel saves go live. The webhook fires immediately on save.

    GitHub Actions workflow (.github/workflows/deploy.yml)

    Triggers on push to main and workflow_dispatch. Steps: checkout → Node 20 → npm ci → vite build (pointing at production Kirby URL) → SamKirkland/FTP-Deploy-Action uploads frontend/build/ to the server webroot.

    Required GitHub Actions secrets

    Secret Value
    KIRBY_URL Production Kirby domain (used by Vite during prerender)
    FTP_SERVER FTP hostname
    FTP_USERNAME FTP username
    FTP_PASSWORD FTP password
    FTP_SERVER_DIR Remote path (e.g. /www/hobbylog.hendrik-berends.de/)

    Kirby webhook hooks (site/config/config.php)

    Three hooks fire on content change: page.update:after, page.create:after, page.delete:after. Each calls triggerDeployment(), which POSTs to the GitHub API workflow_dispatch endpoint. A no-op if deploy.php is absent — local dev is unaffected.

    Server-side credential file

    site/config/deploy.php (gitignored, server-only) holds the GitHub PAT (workflow scope) and repo name. site/config/deploy.example.php is committed as a template.

    Next steps to activate

    • Create the 5 GitHub Actions secrets in repo settings
    • Generate a GitHub PAT with workflow scope
    • Upload deploy.php to site/config/ on the server
    • Merge branch and push — first run validates the pipeline

    ⚖️ Design Decisions

    Why adapter-static and not adapter-node?
    The host provides PHP but no Node.js runtime. The static output can be uploaded via FTP like any other HTML/CSS/JS site.
    Why prerender = true in +layout.ts?
    All routes should be pre-rendered at build time (when Kirby is running). During vite dev, SvelteKit fetches live from Kirby instead — no prerendering happens in dev mode.
    Why handleHttpError: 'warn'?
    If Kirby is offline during a build run, the prerender phase gets a 500. Without this setting the build crashes. With 'warn', it logs and continues.
    Why not use Svelte's native <dialog> element?
    The <dialog> element requires calling showModal() via a DOM reference, which is awkward with Svelte's declarative model. A role="dialog" div with manual keyboard handling is simpler.
    Why Tailwind v4 (Vite plugin) and not v3?
    Tailwind v4 eliminates the tailwind.config.js file and content scan config. The @tailwindcss/vite plugin handles everything automatically via CSS @import 'tailwindcss'.

    ⚠️ Gotchas & Fixes

    Node version conflict (Herd vs Homebrew)

    Laravel Herd installs Node v16 and puts it first on $PATH. Vite 7 and SvelteKit 2 require Node ≥ 20.

    Workaround: Always invoke the Vite binary directly with the Homebrew Node: 

    /opt/homebrew/bin/node node_modules/.bin/vite dev
/opt/homebrew/bin/node node_modules/.bin/vite build

    Svelte a11y: role="dialog" needs tabindex

    Svelte's accessibility linter warns that elements with interactive roles need a tabindex. Added tabindex="-1" to the dialog div.

    Blueprint gap — flat vs nested miniatures

    Two miniatures field blueprints exist: fields/miniatures.yml (flat, targeted by SvelteKit) and fields/structure-miniatures.yml (nested). Existing content uses the nested format, so the KQL query returns empty arrays until blueprint and content are migrated.

    🎨 Paint Progress on Cards

    The collection index grid previously showed only a miniature count and wishlist count per card. A compact paint progress bar was added to each card showing the percentage of owned minis that are painted.

    Implementation

    Done inline in src/routes/+page.svelte using Svelte's @const tag inside the #each block — no additional component import needed: 

    {@const owned = page.miniatures.filter((m) => !m.wished)}
{@const ownedAmt = owned.reduce((s, m) => s + m.amount, 0)}
{@const paintedAmt = owned.filter((m) => m.painted).reduce((s, m) => s + m.amount, 0)}
{@const pct = ownedAmt > 0 ? Math.round((paintedAmt / ownedAmt) * 1000) / 10 : 0}
    Why inline instead of reusing PaintProgress.svelte?
    PaintProgress.svelte has a hardcoded mb-6 margin and a "Bemalfortschritt" label — both are appropriate for the detail page but too heavy for a card. An inline bar avoids modifying the shared component or adding a compact-mode prop.

    Counting method

    Uses amount (sum of miniature counts), consistent with the global stats donuts on the index page. At 100% the bar fills completely green. At 0% it collapses to nothing (no empty label shown).

    Gotcha: SVG presentation attributes and CSS variables

    SVG stroke="{value}" does not evaluate CSS variables. The DonutChart's track circle used stroke={trackColor} (a Svelte attribute binding), which sets the SVG presentation attribute directly — bypassing the CSS cascade entirely. Switching to style="stroke: {trackColor}" routes it through CSS, where var(--chart-track) resolves correctly.

    Gotcha: Svelte class: directive and Tailwind modifiers

    class:dark:text-gray-400={condition} does not work as intended. While Svelte will technically add the class name string dark:text-gray-400 to the element, Tailwind v4 only generates CSS for classes it detects statically during the build scan — a dynamically toggled dark: class may not be included. The safe pattern is to use a computed class string in the template literal: class="{condition ? 'text-blue-600' : 'text-gray-600 dark:text-gray-400'}".

    🔍 Search & Sort

    Search

    A searchQuery state was added to [slug]/+page.svelte and passed to FilterBar as a $bindable prop. The input uses type="search" for native browser clear button support. Filtering matches only name — not original_name or notes — to avoid surprising results.

    Why name-only search?
    Matching original_name caused entries like "Vorbote des Verfalls" to appear when searching "ca" (because the original name contained it), with no visual indication of why. Name-only is predictable and the highlights are always visible on the card.

    Highlighting uses a highlight() helper that HTML-escapes the text, splits on the query with a case-insensitive regex, and wraps matches in <mark> tags with a yellow/dark background. Rendered via {@html} in both MiniatureCard and MiniatureTable.

    Sorting

    A sortBy state ('name' | 'amount' | 'painted') was added alongside the existing filters. Sorting is applied after filtering inside filteredMiniatures, which was upgraded from $derived to $derived.by() to accommodate both steps. Default is name (omitted from URL). The sort dropdown lives in the right-side toolbar next to the view toggle.

    Both searchQuery and sortBy persist in the URL (?q= and ?sort=) using the existing goto() pattern.

    🃏 Card & Table Improvements

    Cards — badges removed, overlay pills added

    The static Painted / Converted / Special badges were removed from the card body. The card now shows only the miniature name below the image — clean by default.

    • ×N (dark translucent, bg-gray-900/70) — always shown (amount >= 1, including single miniatures)
    • Painted (green) / Unpainted (red) — shown when sortBy === 'painted', so you can see status while working through the sort

    Props flow: +page.svelte → MiniatureCards → MiniatureCard. showPainted is the only remaining boolean prop; showAmount was removed once amount became always-on.

    The amount pill uses amount >= 1 — it shows on every card including single miniatures, so there is no ambiguity about whether a card represents one or more.
